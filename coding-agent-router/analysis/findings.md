The pairwise CIs overlap (`all_frontier`'s lower CI 2.8% < `full_system`'s upper CI 16.1%), so

(10% > 0%) and the *direction* of the empty-patch gap (12 > 3 > 0) is consistent, but the
sample is too small to make a confidence claim.

## Cost story (the only thing that *is* statistically clear)

The router achieves **a ~29× cost reduction** vs. always-frontier ($0.025/traj vs. $0.74/traj),
while routing ~40% of requests to the frontier backend. This is the engineering claim that
holds up cleanly. The quality claim does not.

## What actually went wrong with `all_local`

Every `all_local` trajectory shows the same pattern: 1–2 steps, wall time ~95s, an empty patch.
Looking at the model output:

```
"opencode_stdout": "{\"name\": \"bash\", \"arguments\": {\"command\": \"ls astropy/io/ascii/qdp.py\", ...}}"
```

`qwen2.5-coder:14b` (served by Ollama) emits its tool call as **JSON in the content field**
rather than as structured `tool_calls` per the OpenAI chat-completion schema. opencode's
`@ai-sdk/openai-compatible` provider treats this as plain assistant text, finds no tool
invocation, exits the agent loop, and writes an empty diff. The pipeline is wired correctly;
the local model's tool-call serialization is simply incompatible with the agent harness.

This is a known property of base Qwen2.5-Coder models (the Instruct variant fixes it, but
isn't packaged as `qwen2.5-coder:14b` in Ollama's catalog). With a tool-call-fluent local
model (e.g., `functionary-small`, `xLAM-Coder`, or a Qwen2-VL tool-tuned variant), the
all_local baseline would produce edits, and the router would have something to recover.

## Routing decisions for `full_system` (n=111 steps across 20 issues)

| Reason                          | Steps |  % of steps | Backend  |
|---------------------------------|------:|------------:|----------|
| `default_local`                 |    63 |       56.8% | local    |
| `initial_planning`              |    21 |       18.9% | frontier |
| `tool_failed`                   |    17 |       15.3% | frontier |
| `error_in_recent_context`       |     7 |        6.3% | frontier |
| `frontier_rate_limited_fallback`|     3 |        2.7% | local    |

40.5% of `full_system` steps went to frontier — the heuristic is firing as designed. The
breakdown shows it triggers most often on **distress signals** (`tool_failed`,
`error_in_recent_context` = 21.6% combined) and **early-step planning** (18.9%). The
remaining 59.5% are local-first attempts, of which ~half failed silently due to the
content-field tool-call issue described above.

The confidence-gate path (`escalated_*` reasons in `proxy/server.py:74-86`) **never fired**
in `full_system` — meaning `parse_response_quality` always returned good enough quality
even when the local response was actually garbage (free-text tool calls). This is the
second concrete weakness: the gate's quality signal doesn't catch the content-field
tool-call failure mode. Adding a "response has no `tool_calls` but content looks like a
tool call" detector would catch this.

## How to write this up honestly

The story has three defensible claims:

1. **The router system functions correctly.** Per-issue session isolation, heuristic
   routing, fail-soft, and the confidence gate all work end-to-end on three machines
   (validated by the smoke test and the all_frontier run that produced 12 real patches
   and 2 resolved instances).

2. **Hybrid routing achieves the cost reduction it was designed for.** ~29× cheaper
   per trajectory than all-frontier, with ~40% of traffic correctly diverted to local.

3. **Hybrid routing did not preserve quality on this benchmark + this local model.**
   This is dominated by an external factor — `qwen2.5-coder:14b`'s tool-call format
   doesn't match opencode's parser — rather than a flaw in the routing logic itself.

The honest framing is *"the routing layer works; the local backend was the wrong choice
for this agent harness."* Future work: swap to a tool-call-fluent local model, or add a
malformed-tool-call detector to the confidence gate.

## Caveats for the writeup

- **n=20 is small.** Wilson CIs span 16 percentage points even on the 0/20 configs;
  treat all absolute numbers as illustrative, not benchmark-leaderboard results.
- **NIM free-tier throttling.** 50% of NIM requests returned 429 on the `all_frontier`
  run; 8/20 issues got empty patches because opencode couldn't iterate through the
  throttle storm. The headline 2/20 is therefore conservative for the model's actual
  capability — the conditional pass rate among non-empty patches is 2/12 ≈ 17%.
- **Cost rates.** Costs are computed at NVIDIA NIM's published $0.0006/$0.0024 per 1K
  prompt/completion tokens for `qwen3-coder-480b`. If your subscription tier differs
  the absolute numbers shift but the relative comparison is preserved.
- **Inline grading.** The benchmark harness's inline swebench call used bare `python`
  rather than `sys.executable`, so per-issue eval silently failed on every issue.
  Headline pass rates above come from a separate post-hoc grading run using
  `.venv/bin/python -m swebench.harness.run_evaluation`.
